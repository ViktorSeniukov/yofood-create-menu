## Чеклист при анализе любой проблемы

Всегда держи в голове — даже если не упоминаешь явно:

- `null` / `undefined` / пустые массивы
- race conditions в `async` composables
- реактивность Vue: потеря реактивности при деструктуризации `props`
- `watch` vs `watchEffect` — нужный ли триггер
- утечки памяти: незакрытые подписки, таймеры без `onUnmounted`
- TypeScript: `any` не используется, типы API соответствуют реальным данным
- кэширование: устаревшие данные после мутации
- Ant Design: форма не сбрасывается, `rules` не срабатывают
- роутинг: `name` роута не совпадает с `routes.ts`
- импорты: нарушен порядок из `CODE_STYLE.md`

## Процесс анализа

### Шаг 1 — Формализация проблемы

- Кратко: что сломано, где, когда, для кого, как часто
- Классификация: `bug` / `performance` / `security` / `reliability` / `data integrity` / `UX` / `regression`
- Риск: уровень влияния × вероятность

### Шаг 2 — Гипотезы причин

Составь 3–10 гипотез. Группируй по категориям:

- Код / логика компонента
- Composable / реактивность
- Сервис / API-слой
- TypeScript / типы
- Ant Design / UI-библиотека
- Роутер / навигация
- Конфигурация / окружение

Для каждой гипотезы:

```
Гипотеза: ...
За: ...
Против: ...
Проверка: ... (какой файл открыть, что добавить в лог, какой тест написать)
```

### Шаг 3 — Диагностика

- Быстрые проверки (5–15 мин): что посмотреть в первую очередь
- Глубокие проверки: если быстрые не дали ответа
- Инструменты: Vue DevTools, browser DevTools, Vitest, console.error с контекстом

### Шаг 4 — Исправления

Давай минимум 2 варианта:

**Hotfix** — минимальное безопасное изменение, не ломает ничего вокруг  
**Правильное решение** — архитектурно верно, соответствует `CODE_STYLE.md`

Для каждого варианта показывай изменения в формате **до/после** или **diff**.  
Включай: импорты, типы, обработку ошибок, граничные случаи.  
Если меняешь публичный интерфейс компонента (props/emits) — объясни план миграции.

Примеры diff для этого стека:

```typescript
// ДО
const user = ref(null)

// ПОСЛЕ
const user = ref<User | null>(null)
```

```vue
<!-- ДО -->
<script setup>
const { name } = defineProps(['name'])
</script>

<!-- ПОСЛЕ -->
<script setup lang="ts">
interface Props {
  name: string
}
const props = defineProps<Props>()
</script>
```

### Шаг 5 — Защита от регрессий

- Тесты: unit (Vitest) / компонентные (@vue/test-utils) / e2e
- Негативные кейсы и edge cases
- Что добавить в `console.error` для будущей диагностики
- Если баг в сервисе — мок для теста через `vi.mock`

### Шаг 6 — Чеклист качества решения

Перед финальным ответом проверь:

```
✅ Корректность логики
✅ Обработка ошибок (try/catch, error.value)
✅ Нет any — только конкретные типы или unknown
✅ Реактивность не потеряна (нет деструктуризации props)
✅ Нет прямых вызовов fetch/axios в компонентах
✅ Нет логики в шаблоне — вынесено в computed
✅ Стили — через токены темы, не через CSS-переопределение AntD
✅ Импорты в правильном порядке
✅ Есть тест или описание как проверить фикс
```

### Шаг 7 — Финальный самоаудит (обязателен для полного уровня)

Ответь явно:

- Что могло остаться неучтённым?
- Какие баги могут появиться позже (список + вероятность)?
- Что нужно сделать, чтобы поймать их заранее?
- Насколько уверен в решении (%) — и почему именно столько
